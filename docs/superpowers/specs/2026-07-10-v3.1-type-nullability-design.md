# v3.1 — `type` and `nullability` claims via pyright

**Status:** design of record. Sub-project 1 of 3 in the v3.1 roadmap (`docs/v3-design.md` §7).
**Date:** 2026-07-10.

## 1. Context

v3.0-min made one claim type deterministic. `cca-fp-check` classifies a finding into a
`claim_type`, and for `definedness` it shells out to `python -m cca_checks definedness`,
which runs pyright and returns a verdict carrying a real artifact. Everything else falls
back to LLM adjudication.

`definedness` is not the claim type that matters most. The canonical AI-review false
positive — the one this project's README and article both open with — is **"possible null
dereference"** on a value that is guarded three lines up. A type checker settles that
question mechanically. It should not be an LLM's opinion.

## 2. Goal

Extend the existing pyright adapter to two more claim types, `nullability` and `type`,
using the same mechanism, the same artifact-or-UNCERTAIN discipline, and the same graceful
fallback. Produce the generic checker seam that sub-projects 2 (taint/semgrep) and 3
(other languages) will build on.

### Non-goals

- `taint` (semgrep) — sub-project 2.
- Languages other than Python — sub-project 3.
- An auditor-emitted claim schema (`docs/v3-design.md` §3.1) — still deferred; `claim_type`
  stays LLM-inferred.
- Semantic / business-logic claims — these remain LLM-adjudicated by design.

## 3. Design

### 3.1 Rule sets

Three frozen sets in `cca_checks/pyright_check.py`, keyed by claim type:

```python
DEFINEDNESS_RULES = {"reportUndefinedVariable", "reportUnboundVariable", "reportMissingImports"}

NULLABILITY_RULES = {"reportOptionalMemberAccess", "reportOptionalSubscript",
                     "reportOptionalCall", "reportOptionalIterable",
                     "reportOptionalOperand", "reportOptionalContextManager"}

TYPE_RULES = {"reportArgumentType", "reportAssignmentType", "reportReturnType",
              "reportCallIssue", "reportIndexIssue", "reportOperatorIssue",
              "reportAttributeAccessIssue", "reportRedeclaration",
              "reportIncompatibleMethodOverride", "reportIncompatibleVariableOverride",
              "reportGeneralTypeIssues"}

RULES_BY_CLAIM = {"definedness": DEFINEDNESS_RULES,
                  "nullability": NULLABILITY_RULES,
                  "type": TYPE_RULES}
```

`reportGeneralTypeIssues` is the pre-split legacy name; keeping it means an older pyright
still *confirms*. Rule-set drift is only ever allowed to cost us a confirmation, never a
refutation — see §3.2 and §3.3.

The three sets must be pairwise disjoint (asserted by a test), so a claim type maps to
exactly one interpretation of a diagnostic.

### 3.2 Three-way verdict, replacing the two-way

Today: *matching rule at the line → CONFIRMED, otherwise → FALSE_POSITIVE.* That has a
false-negative hole. If pyright renames a rule (it split `reportGeneralTypeIssues` into
granular rules once already), a real diagnostic arrives under a name not in our set, we see
"no match," and we **refute a real bug**.

`verdict_for_claim(claim, diags, rules)` therefore returns:

| pyright state at `claim.line` | verdict | source |
|---|---|---|
| `diags is None` (tool absent) | `UNCERTAIN` — "pyright unavailable; falling back to LLM" | `llm` |
| a diagnostic whose `rule` ∈ `rules` | `CONFIRMED` — rule + message | `pyright` |
| diagnostics present, **none** in `rules` | `UNCERTAIN` — "pyright reported N diagnostic(s) at this line but none in the expected rule set; escalated" | `pyright` |
| no diagnostic at all | `FALSE_POSITIVE` (subject to §3.3) | `pyright` |

Same discipline as the repro runner: **escalate, never false-refute.** It hardens
`definedness` for free. `verdict_for_definedness` becomes a one-line wrapper over
`verdict_for_claim` so nothing that calls it changes.

### 3.3 The blindness probe (the part that makes refutation safe)

`definedness` is **annotation-independent**: pyright resolves a name whether or not
anything in the file is typed. Silence is real evidence.

`nullability` and `type` are **not**. Given `def charge(user):`, `user.card.token` produces
no diagnostic — not because the access is safe, but because pyright knows nothing about
`user`. Refuting a nullability claim there would silently convert a false positive into a
false negative, which is the exact trade this project exists to refuse.

So on the refute path only, and only for `nullability` / `type`:

1. Find the claim line's **enclosing function** via stdlib `ast` (nearest `FunctionDef` /
   `AsyncFunctionDef` whose line span contains the claim line; the module if there is none).
2. Re-run pyright over the same file in **strict** mode, via a generated temporary
   `pyrightconfig.json` (`{"include": [<abs file>], "typeCheckingMode": "strict"}`) passed
   with `--project`. Strict is the only mode that emits the blindness rules.
3. If any diagnostic with a rule in

   ```python
   BLINDNESS_RULES = {"reportMissingParameterType", "reportUnknownParameterType",
                      "reportUnknownMemberType", "reportUnknownVariableType",
                      "reportUnknownArgumentType", "reportMissingTypeStubs"}
   ```

   lands **inside that function's line span**, pyright is blind there → `UNCERTAIN`:
   *"pyright has no type information in the enclosing scope; escalated."*
4. Otherwise the refutation stands: `FALSE_POSITIVE` with the pyright artifact.

The probe is scoped to the enclosing function, not the line, because the blindness
diagnostic fires on the `def` (the untyped parameter), not on the access that dereferences
it. It is scoped to the enclosing function, not the file, because file scope would escalate
everything and buy nothing.

**Fail-safe:** if the probe cannot run for any reason — config write fails, pyright errors,
the file will not parse — return `UNCERTAIN`, never `FALSE_POSITIVE`. Blindness is assumed.

Cost: one extra pyright invocation, only on the refute path, only for two claim types.
Single-file pyright runs are fast, and a refutation is the one verdict worth paying for.

### 3.4 CLI

One generic subcommand:

```
python -m cca_checks check --claim-type {definedness,type,nullability} \
    --finding-id ID --file PATH --line N [--symbol S]
```

`definedness` is retained as a **back-compat alias** that dispatches to the same code path.
This is not ceremony: `cca-fp-check.md` is *copied* into `.claude/agents/` while `cca_checks`
is *pip-installed*. The two can drift — a user updates one and not the other. A three-line
alias makes that mismatch harmless.

Output is unchanged: one line of JSON, `asdict(Verdict)`, exit 0.

### 3.5 Bug fixed in passing

`__main__.py` currently calls `Claim(a.finding_id, a.file, a.line, "definedness", a.symbol)`.
The fifth positional parameter is `proposition`, not a symbol field — `--symbol` is being
misfiled, not merely ignored. Pass it as an explicit keyword `proposition=` and document
the flag as informational: matching is line + rule granular, and `--symbol` never
participates in it.

### 3.6 Deferred minors folded in

- `_diag_at` raises `AttributeError` when a diagnostic carries `"range": None`. Guard with
  `(d.get("range") or {}).get("start") or {}`.
- The artifact-or-UNCERTAIN rule is stated twice in `cca-fp-check.md`. State it once.
- `.gitignore` misses `build/` and `*.egg-info/`, which `pip install .` creates.

## 4. `fp-check` protocol changes

In `claude-code/agents/cca-fp-check.md`, Phase 1 gains two rows in the claim-type table:

- **`nullability`** — the finding asserts a value may be `None`/absent at a use site.
  → `python -m cca_checks check --claim-type nullability --file F --line L`
- **`type`** — the finding asserts a value's type is wrong for its use (bad argument,
  bad return, bad operand, missing attribute).
  → `python -m cca_checks check --claim-type type --file F --line L`

The agent uses the returned JSON **verbatim**. It may not overturn a `CONFIRMED` or a
`FALSE_POSITIVE` that carries a pyright artifact. It adjudicates only `UNCERTAIN`, and when
it does, it must cite the gathered facts and emit `source: llm`.

`crash_impact` keeps its existing rule: unreproduced → `UNCERTAIN`, never `FALSE_POSITIVE`.

## 5. Testing

**Unit** (`tests/test_pyright_check.py`, `tests/test_cli.py`), on synthetic diagnostics:

- the three rule sets are pairwise disjoint;
- matching rule → `CONFIRMED`; unexpected-rule-at-line → `UNCERTAIN`; clean line →
  `FALSE_POSITIVE`; `diags is None` → `UNCERTAIN` + `source: llm`;
- `_diag_at` survives `"range": None`;
- blindness probe: annotated enclosing function → refutation stands; unannotated → `UNCERTAIN`;
  probe raises → `UNCERTAIN`;
- `check --claim-type definedness` and the `definedness` alias produce identical JSON.

**Acceptance** (`tests/acceptance/test_trap_suite.py`, each `skipif shutil.which("pyright") is None`).
Four fixtures, which together are the whole argument:

| fixture | claim | expected |
|---|---|---|
| `guarded_optional.py` — annotated, `if x is None: return` then `x.foo()` | `nullability` @ the access | **`FALSE_POSITIVE`** with a pyright artifact |
| `unguarded_optional.py` — annotated, no guard | `nullability` @ the access | `CONFIRMED`, `reportOptionalMemberAccess` |
| `untyped_optional.py` — **unannotated** param, no guard | `nullability` @ the access | **`UNCERTAIN`** (blind), *never* `FALSE_POSITIVE` |
| `bad_arg_type.py` — call with a wrong argument type | `type` @ the call | `CONFIRMED`, `reportArgumentType` |

Row 1 is the marquee: the article's opening hallucination, refuted mechanically. Row 3 is
the guarantee that row 1 was not bought with a false negative.

## 6. Compatibility

A strict superset of v3.0-min, which was itself a strict superset of v2. Missing pyright,
an unsupported language, or an unmapped claim type all degrade to LLM adjudication with
`source: llm`. No pipeline step gains a hard dependency. `pyproject.toml` version → `0.2.0`.

## 7. Acceptance criteria

1. `python -m cca_checks check --claim-type nullability …` returns a JSON `Verdict`.
2. The four acceptance fixtures produce the four verdicts in §5, with pyright installed.
3. The full suite passes with pyright absent (acceptance rows skip; no failures).
4. `verdict_for_claim` never returns `FALSE_POSITIVE` for a `type`/`nullability` claim
   whose enclosing scope is untyped.
5. `definedness` CLI behaviour is byte-identical to v3.0-min.
