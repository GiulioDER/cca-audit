# v3.2 — `taint` claims via semgrep

**Status:** design of record. Sub-project 2 of 3 in the v3.1 roadmap (`docs/v3-design.md` §7).
**Date:** 2026-07-10.
**Depends on:** v3.1 (`v3.1-type-nullability`, PR #8) — reuses `verdict_for_claim`'s discipline and
`enclosing_span`.

## 1. Context

v3.0 made `definedness` deterministic; v3.1 added `nullability` and `type`. Both lean on pyright,
which is a soundish type checker: within its coverage, its silence is evidence.

`taint` is the third canonical AI-review false positive — *"a SQL injection your ORM already
parameterizes."* The obvious move is to point semgrep at it and treat a hit as `CONFIRMED`, mirroring
the pyright adapter. **That move is wrong, and we can show it is wrong.**

Running semgrep 1.168.0 with a plain taint rule (source: a request-parameter getter; sink: a database
cursor's execute method; sanitizer: an int cast) over four fixtures:

| fixture | semgrep taint rule |
|---|---|
| user input concatenated into SQL | **fires** ✓ |
| a safely parameterized execute, value bound as an argument | **fires** ✗ |
| int-sanitized input | silent ✓ |
| an ORM filter expression | silent ✓ |

Semgrep flags the parameterized query. Taint reaches the sink; that it arrives as a *bound parameter*
rather than as string content is invisible to the pattern. Semgrep is not an analyzer whose silence
or speech constitutes proof — it is a heuristic pattern matcher, and its false-positive rate is the
very thing this project exists to eliminate.

So: a semgrep hit does not prove a bug. Semgrep silence does not prove safety (rule coverage is
unknown). Letting a hit set `CONFIRMED` would stamp "verified by tool" on a false positive, which is
strictly worse than an honest `UNCERTAIN` — it launders a guess into an artifact.

But one question semgrep answers **soundly**, with no dataflow at all: *does a sink of class S occur
in this scope?* That is pure pattern matching. And it is exactly what refutes the ORM case — the
finding asserts a raw-SQL sink; there is none; the premise is false.

## 2. Goal

Let semgrep **refute a taint claim's premise, and inform the LLM's adjudication. Never confirm.**

### Non-goals

- `CONFIRMED` for `taint`. By decision, not by omission. Confirmation stays with the LLM, which now
  adjudicates with a real dataflow trace instead of a vibe.
- Languages other than Python — sub-project 3.
- The semgrep registry (`p/security-audit`) — see §3.1.
- Sink classes beyond the four in §3.2.

## 3. Design

### 3.1 Why a bundled ruleset, not the registry

Refutation is now the *only* verdict semgrep issues, so a refutation is exactly as trustworthy as our
knowledge of what the ruleset covers. Registry rules are fetched over the network at verify time and
change underneath us: the same finding could be refuted today and escalated next week, and we could
never truthfully say "we cover this sink class." A bundled catalog is offline, versioned,
unit-testable, and — decisively — **its coverage is knowable**.

Rules live in `cca_checks/rules/*.yaml`, shipped as package data.

### 3.2 The two-tier sink catalog

Here the danger inverts relative to v3.1. There, the hazard was pyright's silence. Here it is **our
catalog's silence**: the fewer sink patterns we ship, the more often "no sink found" is true, and the
more real bugs we refute. A narrow catalog is a false-negative generator. Coverage must be provable,
not assumed. Hence two tiers, per sink class:

- **STRICT** — vetted sink patterns. A match means *the sink exists*.
- **LOOSE** — name heuristics for sinks we have not vetted. A match means *we may be looking at a sink
  we do not recognize*, and we must not refute.

Supported classes for Python, `SUPPORTED_SINK_CLASSES`. Sinks are described here by name; the semgrep
pattern bodies live in the rule files and are the implementation's business.

| class | STRICT — vetted sinks | LOOSE — unvetted name heuristics |
|---|---|---|
| `sql` | a database cursor's `execute` and `executemany` methods | attribute calls named `sql`, `raw`, `executescript`, `execute_query` |
| `command` | the `system` and `popen` functions of `os`; `subprocess` `run` / `call` when `shell` is true | attribute calls named `run`, `spawn`, `shell` |
| `code_exec` | the two dynamic-evaluation builtins; the unsafe-deserialization loader of the standard serialization module | attribute calls named `eval`, `loads` |
| `path` | the file-opening builtin; `os.remove`; the recursive-tree remover in `shutil` | attribute calls named `read_text`, `write_text`, `unlink` |

The LOOSE tier is the anti-false-negative guarantee and is the structural analogue of v3.1's blindness
probe: **suspicious-but-unmatched escalates; only provably-absent refutes.**

What this spec fixes is the tiering, the verdict table, and the guarantee that a LOOSE match can never
be refuted away. Adding a vetted sink to STRICT, or a heuristic to LOOSE, is a rule-file change.

### 3.3 The verdict, scoped to the enclosing function

Scoping matters, and line-exactness would be a bug. In the vulnerable fixture the sink call sits one
line below the string concatenation an auditor would flag. A line-exact "no sink here" test would
refute a real bug. So the question asked is always: *does a sink of class S occur anywhere in the
claim line's enclosing function?* (module scope if the line is top-level).

`verdict_for_taint(claim, ...)` returns:

| state | verdict | source |
|---|---|---|
| semgrep missing / errored / timed out / scanned 0 files / unparseable output | `UNCERTAIN` — "semgrep unavailable; falling back to LLM" | `llm` |
| `claim.sink_class` not in `SUPPORTED_SINK_CLASSES`, or the file is not Python | `UNCERTAIN` — "sink class / language not covered; escalated" | `llm` |
| a **STRICT** sink occurs in the enclosing scope | `UNCERTAIN` — sink located, plus the taint-rule match if one was found (§3.4) | `semgrep` |
| no STRICT sink, but a **LOOSE** near-miss occurs in the enclosing scope | `UNCERTAIN` — "possible unrecognized sink; escalated" | `semgrep` |
| neither | **`FALSE_POSITIVE`** — "no `<class>` sink in the enclosing scope @ file:line; the finding's premise does not hold" | `semgrep` |

`CONFIRMED` is unreachable for `taint`. A test asserts this over the whole fixture corpus.

### 3.4 Informing the adjudication

When a STRICT sink is present, also run the bundled taint rule for that class and, if it matched,
embed a compact summary in the `UNCERTAIN` evidence: **rule id, sink line, and the rule's message**.

Measured constraint, not an assumption: semgrep OSS 1.168.0 does **not** populate
`extra.dataflow_trace` for these rules (it stays `null` even with `--dataflow-traces`), and
`extra.lines` returns the placeholder `"requires login"` rather than the matched source. So the
evidence is the rule match — id, location, message — not a source-to-sink trace and not a code
snippet. Do not design around a trace we cannot obtain.

The `fp-check` agent must treat this as **gathered facts**, not a verdict — it may adjudicate
`CONFIRMED` from it, but the adjudication and the `source: llm` are its own.

### 3.5 The semgrep runner

`run_semgrep(config_path, file) -> Optional[list[dict]]` mirrors `run_pyright`'s fail-safe cascade
exactly, and for the same reason:

- `FileNotFoundError` / `TimeoutExpired` / `OSError` → `None`
- empty or unparseable stdout → `None`
- non-empty `errors[]` in the JSON (parse failure, rule error) → `None`
- fewer than one entry in `paths.scanned` → `None`
- otherwise → `results` (possibly `[]`)

`None` means "could not tell — escalate". `[]` means "semgrep ran and matched nothing". Conflating
them is precisely the bug that was found and fixed in `run_pyright` during v3.1 review.

Invocation is offline and reproducible — the bundled config, JSON output, metrics off, version check
disabled, quiet, over a single file — with `timeout=120`. No network at verify time.

One measured detail with teeth: **semgrep namespaces `check_id` by the config file's path.** A rule
declared `id: sink-strict-sql` inside `cca_checks/rules/python_sinks.yaml` is reported as
`cca_checks.rules.sink-strict-sql`. Rule matching must compare the **last dot-separated segment**,
never the whole string. A unit test built on bare ids passes while the real integration matches
nothing and every claim silently escalates.

### 3.6 Shared scope helper

`enclosing_span(path, line)` currently lives in `cca_checks/pyright_check.py` and is now needed by two
adapters. Move it to a new `cca_checks/scope.py`; `pyright_check` imports it from there so nothing
that calls it changes. This is the one refactor in scope, and it is the natural seam — v3.3 (other
languages) will need a non-`ast` implementation behind the same interface.

### 3.7 Claim and CLI

`Claim` gains one optional field, `sink_class: str = ""` (frozen dataclass, defaulted, so every
existing construction keeps working).

```
python -m cca_checks check --claim-type taint --sink-class sql \
    --finding-id ID --file PATH --line N
```

`--sink-class` is **not** an argparse `choices` list. An agent may legitimately name a class we do not
cover, and the correct response is `UNCERTAIN`, not a non-zero exit and a traceback. Unknown class →
escalate. A `taint` claim with no `--sink-class` → `UNCERTAIN` ("no sink class given"), never a
refutation.

Output is unchanged: one line of JSON, `asdict(Verdict)`, exit 0.

## 4. `fp-check` protocol changes

`claude-code/agents/cca-fp-check.md` gains one Phase-1 row:

- **`taint`** — the finding asserts untrusted input reaches a dangerous sink. Classify the sink into
  one of `sql`, `command`, `code_exec`, `path`; if it fits none, do not pass `--sink-class`.
  `python -m cca_checks check --claim-type taint --sink-class <CLASS> --finding-id <ID> --file <FILE> --line <LINE>`

And one instruction, stated once:

> A `taint` verdict is never `CONFIRMED` by the checker. A `FALSE_POSITIVE` means no sink of that
> class exists in the enclosing scope and the finding's premise does not hold — honour it. An
> `UNCERTAIN` carrying a taint-rule match means semgrep found a source-to-sink path **but cannot tell
> a real injection from a safely-parameterized call** — you must adjudicate, citing the match and the
> code you read, and emit `source: llm`. Do not treat the match as proof.

The existing rules are untouched: artifact-or-UNCERTAIN; no overturning an artifact-backed verdict;
`crash_impact` unreproduced → `UNCERTAIN`; tool absent → `source: llm`.

## 5. Testing

**Unit** (`tests/test_semgrep_check.py`, `tests/test_scope.py`, `tests/test_cli.py`), on synthetic
semgrep JSON:

- `run_semgrep` fail-safe cascade: each of missing binary, timeout, `OSError`, empty stdout,
  unparseable stdout, non-empty `errors[]`, zero files scanned → `None`; a genuine clean run → `[]`
  (both directions pinned, or a lazy `return None` would pass);
- STRICT hit in span → `UNCERTAIN`; LOOSE-only hit in span → `UNCERTAIN` with the near-miss wording;
  neither → `FALSE_POSITIVE`;
- a STRICT hit **outside** the enclosing span does not prevent a refutation, and a LOOSE hit outside
  the span does not force an escalation;
- unknown `sink_class`, empty `sink_class`, non-Python file → `UNCERTAIN` / `source: llm`;
- `verdict_for_taint` never returns `CONFIRMED`, for any input;
- `enclosing_span` still behaves identically after the move (its v3.1 tests move to `test_scope.py`).

**Acceptance** (`tests/acceptance/test_taint_suite.py`, each skipped when semgrep is not on `PATH`).
Four fixtures, verified against real semgrep 1.168.0 during design:

| fixture | claim | expected |
|---|---|---|
| `orm_no_sink.py` — an ORM filter expression, no raw-SQL sink | `taint/sql` | **`FALSE_POSITIVE`** with a semgrep artifact |
| `sql_concat.py` — user input concatenated, then executed | `taint/sql` | `UNCERTAIN`, evidence names the taint rule and sink line, **never** `FALSE_POSITIVE` |
| `sql_parameterized.py` — value bound as a query argument | `taint/sql` | `UNCERTAIN` — neither blessed nor refuted |
| `sql_unlisted_driver.py` — an unvetted driver method, LOOSE-only match | `taint/sql` | `UNCERTAIN` (LOOSE), **never** `FALSE_POSITIVE` |

Row 1 is the marquee. Row 3 proves we do not launder semgrep's own false positive into a verdict.
Row 4 proves row 1 was not bought with a false negative.

## 6. Compatibility

A strict superset of v3.1. semgrep absent, a non-Python file, an uncovered sink class, or a missing
`--sink-class` all degrade to LLM adjudication with `source: llm`. No pipeline step gains a hard
dependency; `cca_checks` still has no runtime Python dependencies (semgrep is an external CLI, exactly
like pyright). `pyproject.toml` → `0.3.0`, and ships `cca_checks/rules/*.yaml` as package data
(resolved at runtime via `importlib.resources`, so it works from an installed wheel). The installers
gain a line noting `semgrep` as an optional tool alongside `pyright` and `pytest`.

## 7. Acceptance criteria

1. `python -m cca_checks check --claim-type taint --sink-class sql …` returns a JSON `Verdict`.
2. The four acceptance fixtures produce the four verdicts in §5, with semgrep installed.
3. The full suite passes with semgrep absent (taint acceptance rows skip) and with pyright absent.
4. `verdict_for_taint` never returns `CONFIRMED`.
5. `verdict_for_taint` never returns `FALSE_POSITIVE` when a STRICT or LOOSE pattern matches inside
   the enclosing scope, when the sink class is uncovered, or when `run_semgrep` returned `None`.
6. No network access at verify time.
7. `definedness` / `nullability` / `type` behaviour is byte-identical to v3.1.
